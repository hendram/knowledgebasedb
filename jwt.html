Deep Explanation: How JWT Signing & Verification Works

1. What JWT Really Is

JWT (JSON Web Token) is a digitally signed message format. It is not encrypted by default. Anyone can read the payload. The security comes from the cryptographic signature.

The signature guarantees:

  • Integrity – the message was not modified
  • Authenticity – the issuer really created it

2. JWT Structure

HEADER.PAYLOAD.SIGNATURE

Each part is Base64URL encoded.

3. Header Example

{
  "alg": "RS256",
  "typ": "JWT"
}

RS256 means:

  • RSA algorithm
  • SHA-256 hashing
  • Asymmetric cryptography (private key signs, public key verifies)

4. Payload Example

{
  "sub": "123456",
  "role": "admin",
  "iat": 1710000000,
  "exp": 1710003600
}

The payload is NOT secret. It is only protected against tampering.

5. How The Signature Is Formed (Mathematical View)

Step 1 — Encode

encodedHeader  = Base64UrlEncode(header)
encodedPayload = Base64UrlEncode(payload)

Step 2 — Create Signing Input

signingInput = encodedHeader + "." + encodedPayload

Step 3 — Hash

hash = SHA256(signingInput)

Step 4 — Sign With Private Key

signature = RSA_sign(hash, privateKey)

Step 5 — Encode Signature

encodedSignature = Base64UrlEncode(signature)

Final JWT:

encodedHeader.encodedPayload.encodedSignature

6. Why The Private Key Is NOT Inside The Signature

The private key is never embedded in the token. It is used only to mathematically generate the signature.

Think of it like this:

  • The private key is a secret stamping tool.
  • The signature is the stamp impression on the paper.

When you see a wax seal from a king, the seal does not contain the king's ring. It contains the impression made by the ring.

You cannot recreate the ring from the wax impression. But you can verify the seal using a known reference of the king's seal (public key).

Mathematically, RSA signing is a one-way operation: 

signature = hash^d mod n

Where:

  • d = private exponent (secret)
  • n = modulus

The public key contains (e, n). It cannot derive d.

7. How Verification Works

The receiver performs: 

recomputedHash = SHA256(signingInput)
verifiedHash   = RSA_verify(signature, publicKey)

Internally, RSA verification does: 

hash = signature^e mod n

If the recomputed hash equals the verified hash → signature valid.

If the payload was changed by even 1 character, the hash changes completely, and verification fails.

8. King Stamp Analogy

Imagine:

  • The king writes a decree (payload).
  • He presses his royal ring (private key) into hot wax.
  • The wax seal is the signature.

Citizens:

  • Have an official drawing of the ring (public key).
  • They compare the seal against the known ring pattern.

If it matches → decree is authentic. If someone forges it without the real ring → verification fails.

The ring never leaves the king. The wax seal does.

9. Security Properties

  • Private key stays only on the issuing server
  • Public key can be distributed to many services
  • Signature cannot be reversed to obtain private key
  • Any tampering breaks verification

10. Important Clarification

JWT with RS256 provides integrity and authenticity, not confidentiality. If you need secrecy, you must encrypt the token (JWE) or use HTTPS.

Conclusion

A JWT signature is not a container of the private key. It is the mathematical result of signing a hash using that private key. The public key allows verification but cannot recreate the private key.
