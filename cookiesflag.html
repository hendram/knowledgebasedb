🍪 Complete Cookie Security Flags Guide

1️⃣ HttpOnly

Stops JavaScript from reading cookie.

Protects against: XSS token theft

Use when: cookie stores auth/session/refresh token

HttpOnly

2️⃣ Secure

Cookie only sent over HTTPS.

Protects against: network sniffing, downgrade attacks

Secure

Never deploy auth cookies without this in production.

3️⃣ SameSite

Controls cross-site sending.

Protects against: CSRF

Value Behavior Use Case
Strict Never cross-site Banking / Admin panels
Lax Top-level navigation only Most applications
None Always send Cross-domain authentication

SameSite=Strict

4️⃣ Path

Limits where cookie is sent.

Example:

Path=/api

Cookie not sent to:

/public

Reduces attack surface.

5️⃣ Domain

Controls which subdomains receive cookie.

Example:

Domain=example.com

Sent to:

  • api.example.com
  • admin.example.com

If omitted → only exact domain.

Best practice: Don’t set Domain unless subdomain sharing is required.

6️⃣ Max-Age / Expires

Controls lifetime.

Short lifetime reduces risk if stolen.

Max-Age=900

= 15 minutes.

7️⃣ Priority (Modern Browsers)

Hints browser which cookies to keep under limits.

Priority=High

Useful for auth cookies so they aren’t evicted.

8️⃣ Partitioned (Advanced)

Prevents third-party cookie tracking abuse.

Mainly used for embedded widgets / iframes.

Not needed for most authentication systems yet.

🏆 Production-Grade Example

Recommended refresh token cookie configuration:

Set-Cookie:
refresh_token=xyz;
Path=/;
HttpOnly;
Secure;
SameSite=Strict;
Max-Age=604800

This configuration is considered a strong baseline security standard.
